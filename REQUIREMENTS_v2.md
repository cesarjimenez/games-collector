# Videogame Collection Service — Requirements Specification v2

---

## 1. Context & Objective

The service is an HTTP API that allows users to manage a personal videogame collection.
It is written in Java, exposes a JSON interface, and uses a file-based database.
The service must be runnable locally without external infrastructure.

There are two user roles:

| Role | Responsibility |
|---|---|
| **Customer** | Tracks their personal game collection |
| **Admin** | Maintains the platform catalogue, game catalogue, and customer registry |

---

## 2. Authentication & Security

### 2.1 Customer authentication

- Every customer request **MUST** include an HTTP header: `Authorization: ApiKey <key>`
- The service validates the key against the database on every request.
- If the header is absent or the key is not found → **401 Unauthorized** (see COMMON.A).
- Customers **CANNOT** invoke admin endpoints.
- A customer **CANNOT** read or modify another customer's collection.

### 2.2 Admin authentication

- Every admin request **MUST** include HTTP Basic Authentication (`Authorization: Basic <base64(user:pass)>`).
- The username and password are fixed values loaded from a configuration file at startup.
- If the header is absent, or the credentials do not match the configured values → **401 Unauthorized** (see COMMON.B).
- Admins **CANNOT** invoke customer endpoints.

### 2.3 TLS

No TLS is required. The service operates over plain HTTP.

## 3. Data Model

### 3.1 Customer

| Field | Type | Constraints |
|---|---|---|
| `username` | String | Unique, mandatory, max 30 chars. Acts as the primary key. |
| `apiKey` | String | Unique, mandatory. Auto-generated by the system on creation. Never provided by the caller. |

### 3.2 Platform

| Field | Type | Constraints |
|---|---|---|
| `id` | String | Unique, mandatory, max 30 chars. Acts as the primary key. Provided by the admin on creation. |
| `year` | Integer | Mandatory. |
| `description` | String | Optional, max 200 chars. |

### 3.3 Game

| Field | Type | Constraints |
|---|---|---|
| `id` | String | Mandatory, max 30 chars. |
| `platformId` | String (FK → Platform.id) | Mandatory. The `(id, platformId)` pair must be globally unique. |
| `title` | String | Mandatory, max 50 chars. |
| `description` | String | Optional, max 200 chars. |
| `genre` | Enum | Mandatory. Fixed set of values: `shmup`, `racing`, `platforms`. |

> **Note on Game identity:** The `id` field alone is **not** unique. Two games may share the same `id` if they belong to different platforms (e.g., `id=sonic` on `platformId=megadrive` and `id=sonic` on `platformId=gamegear` are two distinct, valid records). The composite key `(id, platformId)` is the unique identifier for a Game.

### 3.4 Collection

| Field | Type | Constraints |
|---|---|---|
| `customerId` | String (FK → Customer.username) | Unique. Acts as the primary key. One collection per customer. |
| `games` | List of `(gameId, platformId)` pairs | References Game records. May be empty. No duplicates. |

### 3.5 Entity relationships

| Relationship | Cardinality |
|---|---|
| Platform → Games | 0..N (one platform may have many games) |
| Game → Platform | 1..1 (one game belongs to exactly one platform) |
| Customer → Collection | 1..1 (each customer has exactly one collection, created automatically) |
| Collection → Games | 0..N (a collection may contain zero or more games) |
| Game → Collections | 0..N (a game may appear in zero or more collections) |

## 4. Functional Requirements

### REQ-001 — List available games (Customer)

**Endpoint:** `GET /games`
**Auth:** Customer (ApiKey)

**Query parameters:**

| Parameter | Type | Required | Description |
|---|---|---|---|
| `platformId` | String | No | Filter by platform ID. |
| `genre` | Enum | No | Filter by genre (`shmup`, `racing`, `platforms`). |
| `cursor` | String | No | Pagination cursor returned by a previous response. Omit for the first page. |

**Behaviour:**

- Returns all games in the database that match the provided filters.
- Filters are combined with AND logic: if both `platformId` and `genre` are provided, only games matching both are returned.
- Results are sorted **alphabetically by `title`** (ascending, case-insensitive).
- Results are paginated using cursor-based pagination.

**Success response:** `200 OK`

```json
{
  "data": [
    {
      "id": "sonic",
      "platformId": "megadrive",
      "title": "Sonic the Hedgehog",
      "description": "...",
      "genre": "platforms"
    }
  ],
  "nextCursor": "<opaque_string_or_null>",
  "total": 42
}
```

- `nextCursor` is `null` when there are no further pages.
- `total` is the total count of matching games across all pages (not just the current page).

### REQ-002 — Add a game to the collection (Customer)

**Endpoint:** `POST /collection/{gameId}/{platformId}`
**Auth:** Customer (ApiKey)

> **Note:** The path must carry both `gameId` and `platformId` because a game is uniquely identified by the composite key `(gameId, platformId)`.

**Behaviour:**

- Adds the identified game to the authenticated customer's collection.
- The operation is **idempotent**: if the game is already in the collection, the request succeeds silently with no state change.

**Success response:** `204 No Content` (empty body)

**Error cases:**

| Code | Condition | Reference |
|---|---|---|
| 401 | Missing or invalid API key | COMMON.A |
| 422 | The `(gameId, platformId)` pair does not exist in the database | REQ-002.A |

---

### REQ-003 — Retrieve own collection (Customer)

**Endpoint:** `GET /collection`
**Auth:** Customer (ApiKey)

**Behaviour:**

- Returns all games in the authenticated customer's collection.
- The list is sorted **alphabetically by `title`** (ascending, case-insensitive).
- No pagination is applied to the collection.

**Success response:** `200 OK`

```json
{
  "data": [
    {
      "id": "sonic",
      "platformId": "megadrive",
      "title": "Sonic the Hedgehog",
      "description": "...",
      "genre": "platforms"
    }
  ]
}
```

**Error cases:**

| Code | Condition | Reference |
|---|---|---|
| 401 | Missing or invalid API key | COMMON.A |

---

### REQ-004 — Create a customer (Admin)

**Endpoint:** `POST /customers`
**Auth:** Admin (Basic Auth)

**Request body:**

```json
{
  "username": "alice"
}
```

**Behaviour:**

- Stores the new customer in the database.
- Auto-generates a unique API key for the customer.
- Automatically creates an empty collection for the new customer.
- The operation is **NOT idempotent**: creating the same username twice is an error.

**Success response:** `201 Created`

```json
{
  "username": "alice",
  "apiKey": "<generated_key>"
}
```

> **Note:** The HTTP status for REQ-004 is `201 Created` (not `204`) because the response body carries the generated API key that the admin must communicate to the customer.

**Error cases:**

| Code | Condition | Reference |
|---|---|---|
| 401 | Missing or invalid admin credentials | COMMON.B |
| 409 | A customer with that `username` already exists | REQ-004.A |

### REQ-006 — Create a game (Admin)

**Endpoint:** `POST /games`
**Auth:** Admin (Basic Auth)

**Request body:**

```json
{
  "id": "sonic",
  "platformId": "megadrive",
  "title": "Sonic the Hedgehog",
  "description": "A fast platformer.",
  "genre": "platforms"
}
```

**Behaviour:**

- Stores the new game in the database.
- The operation is **NOT idempotent**: creating the same `(id, platformId)` pair twice is an error.

**Success response:** `204 No Content` (empty body)

**Error cases:**

| Code | Condition | Reference |
|---|---|---|
| 401 | Missing or invalid admin credentials | COMMON.B |
| 409 | A game with the same `(id, platformId)` pair already exists | REQ-006.A |
| 422 | The provided `platformId` does not exist in the database | REQ-006.B |
| 422 | The provided `genre` is not one of `shmup`, `racing`, `platforms` | REQ-006.C |

## 5. Error Response Format

All error responses with a body use the following JSON structure:

```json
{
  "error": {
    "code": "CUSTOMER_ALREADY_EXISTS",
    "message": "A customer with username 'alice' already exists."
  }
}
```

### Error code catalogue

| Reference | HTTP Status | `error.code` | Description |
|---|---|---|---|
| COMMON.A | 401 | `INVALID_API_KEY` | API key header missing or key not found. |
| COMMON.B | 401 | `INVALID_ADMIN_CREDENTIALS` | Basic auth header missing or credentials incorrect. |
| REQ-002.A | 422 | `GAME_NOT_FOUND` | The `(gameId, platformId)` composite key does not exist. |
| REQ-004.A | 409 | `CUSTOMER_ALREADY_EXISTS` | A customer with that username already exists. |
| REQ-005.A | 409 | `PLATFORM_ALREADY_EXISTS` | A platform with that ID already exists. |
| REQ-006.A | 409 | `GAME_ALREADY_EXISTS` | A game with that `(id, platformId)` pair already exists. |
| REQ-006.B | 422 | `PLATFORM_NOT_FOUND` | The referenced platform does not exist. |
| REQ-006.C | 422 | `INVALID_GENRE` | The genre value is not in the allowed set. |

## 6. Technical Constraints

### 6.1 Language & runtime
- Implemented in **Java**.
- Must run locally without external services (file-based database only).

### 6.2 Architecture
- Must follow **Hexagonal Architecture** (Ports & Adapters).
- The domain layer must have zero dependencies on HTTP or persistence frameworks.

### 6.3 Code quality
- Follow SOLID principles.
- Apply software design patterns where they reduce complexity; do not apply them speculatively.
- No code duplication. Prefer composition over inheritance.
- No dependency cycles between modules.
- Cyclomatic complexity per class: **≤ 8**.
- Target algorithm complexity: **O(N)**. Any deviation must be documented with a comment in the source code.

### 6.4 HTTP interface
- All request and response bodies are **JSON**.
- No TLS required.
